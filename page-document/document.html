About this project Actively maintained

A personal R&D project on agent-first software delivery

I built this kit to answer a question I kept hitting in real work: how do you make AI coding agents behave like disciplined engineers instead of confident chat partners? The result is a 16-skill harness that anchors every change to a locked spec, plans tasks in 2–5 minute units, gates completion behind mechanical verification, and grows a durable memory tier over time so the next session doesn't start from zero.

Everything on this page — the lifecycle, the verify gate, the memory tiers, the failure taxonomy — is implemented in the repo and shipped as installable skills you can drop into any codebase. The diagrams render live from documents/diagrams/*.mmd — the same source of truth used by the kit itself.

Start in 10 Minutes

This is the minimum safe path for a team that wants to trial the kit without reading the entire document first.

1. Pick the target repo From the kit repository, decide which project path will receive the harness files.
2. Install the kit Run curl -fsSL https://raw.githubusercontent.com/thaihai-swe/CoreZero-Nexus/main/scripts/install.sh | bash -s -- /path/to/project. The installer applies manifest.json: it copies skills/ and rules/, and seeds the memory router and starter memory.
3. Run /starter-init Bootstrap the harness: capture stack and operating defaults in core-policies.md, seed the memory tiers, and run a clean-state check.
4. Create durable context Run /context-memory and refine the starter memory files, including the split AI coding contract in repo memory.
5. Start the first feature Run /spec-requirements with a real feature slug and lock the acceptance criteria.
Success looks like: a real feature has spec.md, a traceable task breakdown, bounded implementation, and fresh verification evidence before merge.

Priority Rules

No flattery, no filler. Start with the answer, action, blocker, or decision. You MUST NOT add ceremonial openers.
Correct false premises. If the user’s premise is wrong, you MUST say so before continuing.
Never fabricate. You MUST NOT invent file paths, commit hashes, test results, API names, library functions, or repository behavior.
Unknown stays unknown. When information is unavailable, you MUST mark it explicitly as [UNKNOWN]. Do not fill gaps with guesses.
Ask only when needed. SHOULD ask before proceeding when ambiguity materially changes the result; otherwise resolve by inspecting the repo.
Touch only the request. Every changed line MUST directly support the user’s request. No drive-by refactors or formatting churn.
Fail loud. You MUST NOT mark work complete if verification was skipped, failed, or only partially run. State exactly what was and was not verified.
Preserve behavior. Existing observable behavior is a contract. You MUST NOT change it unless the user explicitly asks.
Keep design SOLID. SHOULD follow SOLID principles when changing or adding software design.
Apply code-design rules. Check docs/policies/code-design.md for cross-cutting pitfalls (hidden coupling, swallowed errors, etc.).
Read the Master Index. Before non-trivial work, SHOULD consult MASTER_INDEX.md at the repo root for context routing and key memory files (such as memories/repo/core-policies.md which declares active status and limits).

Use this kit when you want AI coding to behave like a disciplined engineering workflow instead of a chat session with weak memory and weak verification.
Problem Ad hoc AI coding drifts because intent, decisions, and proof are scattered across chat history.
Mechanism The kit anchors work to skills, durable artifacts, and explicit verification rules.
Outcome Teams get cleaner reviews, resumable sessions, and less ambiguity between request and code.

Solving the Agent Drift problem by anchoring every action to durable, verified artifacts.

What Problem This Kit Solves

  • Agent Drift: AI implementation diverges from real user intent.
  • Context Loss: long sessions lose rationale and decisions.
  • Weak Verifiability: changes look plausible but are not evidence-backed.

What Users Get

  • Clear intent: requirements are locked before implementation.
  • Traceability: requirement -> task -> validation remains explicit.
  • Durability: artifacts survive chat/session resets.
  • Operational discipline: completion requires fresh evidence, not assumptions.

Direct Benefits for Customers

  • Zero Slop Code: Code is written strictly in response to planned, small micro-tasks. No bloated, unused, or speculative code reaches the codebase.
  • Clear Traceability: Every line of committed code is traceable back to a micro-task, a planned design, and an acceptance criteria (AC-*) approved during the specification stage.
  • Continuous Safety: Integrated verification gates ensure that existing tests and compilation checks must pass at 100% on the agent's machine before code submission.
  • Predictable Velocity: Socratic requirements grilling flags ambiguities and structural decisions early, avoiding costly rework in the middle of implementation.
  • Durable Knowledge Base: The agent reads the codebase's constitution, policies, and domain glossaries dynamically, ensuring it respects existing architectural patterns and vocabulary.

3. System Architecture

CoreZero standardizes developer actions into structured artifacts. Below is the System Context showing how the developer, the agent runtime, and the repository memory databases interact:

A. The Skill Layer (Logic)

Located in skills/. Defines the Engineering Contract—the "Brain" of the workflow telling agents what to read, how to operate, and when to stop.

B. The Entrypoint Layer

Root entrypoints (AGENTS.md, etc.). A Thin Wrapper that enforces the "Skill-First" rule and operating principles such as goal framing and search-before-building.

C. The Harness Layer (Environment)

Implements the 7-pillar ETCLOVG Taxonomy: Execution, Tools, Context, Logic, Observability, Verification, and Governance. Controls the agent runtime environment and establishes structural guardrails.

D. The Artifact Layer (State)

Located in artifacts/features/<slug>/. The Source of Truth—a durable record of decisions surviving session resets and window limits.

E. The Memory Layer (Durable Guidance)

Located in memories/repo/. Organized into three tiers behind a routing index (MASTER_INDEX.md). The knowledge base operates as a Semantic Memory Knowledge Graph using Markdown links to bind heuristics to domain specs. The tiers are: the Instruction tier (human-curated rules and knowledge), the Auto tier (failure-driven harness-telemetry.md), and the Extracted tier (per-feature session-extracts.md candidates).

Three Memory Tiers:

  • Instruction tier (human-curated): durable rules and knowledge — core-policies.md, project-knowledge-base.md, learned-heuristics.md, and docs/project/architecture.md. Edited by humans and /context-memory.
  • Auto tier (agent-written): harness-telemetry.md — failure-driven lessons appended by /harness-maintain Improve Mode and /harness-verify.
  • Extracted tier (session distillation): artifacts/features/<slug>/session-extracts.md — candidate lessons written by /context-session END and the post-ship sync, triaged by /context-memory.

Flow: session extracts → triage → instruction tier (or discard), with the auto tier capturing failures along the way. The memory router MASTER_INDEX.md decides which files a session loads by matching the task against trigger keywords.

Core Design Principles

1. Artifact-First If it isn't in the artifact, it didn't happen. Chat is transient; artifacts are durable memory.
2. Surgicality Every change must trace to a specific Task ID to prevent drive-by refactoring.
3. Subagent-Driven Use subagents for heavy exploration to keep the main context loop lean and high-signal.
4. Continuity-First Mandatory handoffs before session ends to solve the context window amnesia problem.
5. Evidence-Based Verification "Done" requires fresh logs, tests, or proof. We judge by results, not vibes.

Memory Promotion Rules

Two promotion paths feed durable memory: manual (via /context-memory) and automatic (the post-ship sync after every passing verify).

  • Cross-Feature Pattern Surfacing: Heuristics track a recurrence-count. Promote a finding into the instruction tier only when it is durable, evidence-backed, and useful beyond one feature. If an execution pattern hits 3+ features, the system automatically drafts a CC-* proposal to promote it to the Constitution; a security finding escalates immediately.
  • Extraction triage: /context-session END appends candidates to session-extracts.md. /context-memory then classifies each candidate: promote (move to instruction tier when durable and evidence-backed), defer (needs more evidence from additional features), or discard (not durable, with reason recorded — never silently delete). This triage process ensures only high-signal lessons compound into durable memory.
  • Post-ship sync: a passing /harness-verify verdict triggers a mandatory memory sweep. Every file in MASTER_INDEX.md is either updated or named with a one-line reason it was untouched. A generic "no updates needed" fails the gate.
  • Promotion thresholds: a memory file at ≥800 lines, 3+ distinct subtopics, or 5+ artifact references is added to the MASTER_INDEX.md promotion watchlist with a split proposal — it is never auto-split.
  • Keep feature-local notes, temporary debugging context, and one-off rationale inside feature artifacts.
  • Do not create competing durable memory files such as AI_CONTRACT.md or DECISIONS.md unless the repository intentionally changes the canonical model.

Agentic Data Flow

User Request → Entrypoint → Skill Contract → Artifacts → Implementation → Verification → Memory Sync

The kit sits between the request and the code change. The entrypoint routes work into a skill, the skill reads and writes artifacts, and the implementation is judged against those artifacts instead of chat history.

Core Policies

The memories/repo/core-policies.md file stores the repo-wide constitution and active session limits, rigor settings, telemetry constraints, and known limits. Read it at session start and before non-trivial work; if reality diverges from the policies, fix one of them.

Memory Promotion Thresholds

Memory grows under quotas, not free expansion. /context-memory uses these thresholds to keep durable knowledge useful instead of bloating context every session.

Trigger Threshold Action
File line count (warn) ≥ 800 lines Flag the file for split during the next sweep.
File line count (force split) ≥ 1200 lines Split into topic files; update MASTER_INDEX.md.
Distinct subtopics in one file ≥ 3 subtopics Promote each subtopic to its own memory file.
Cross-references to one entry ≥ 5 references Promote candidate to durable memory tier.

Below threshold, candidates stay in session-extracts.md awaiting more evidence. The kit prefers proven patterns over speculative ones.

Subagent Fan-Out

Heavy reads (codebase exploration, multi-file research, hallucination cross-checks) are delegated to subagents so the main context stays clean. Subagents return a structured summary, not raw output; only the summary enters the parent context.

Failure Classification

When something goes wrong, classify the cause before patching. Each class routes to a different fix: harness changes, model retries, or spec rework.

Harness Problem

The kit's rails let the agent off the rails — missing gate, stale memory, wrong skill routing.

Fix: update skills, memory, or verification gates so the same class of failure can't recur.

Model Problem

The agent had everything it needed but still produced a flawed output — reasoning slip, fabrication, or capability gap.

Fix: tighten prompts, add adversarial pass, or re-route to a more capable model for that phase.

Spec Problem

The requirements were ambiguous, contradictory, or missing — agent implemented the wrong thing correctly.

Fix: return to /spec-requirements, run another grilling wave, lock open decisions.

Common Feedback Loops

  • Discovery loop: implementation uncovers unknown behavior, so work moves back to research or planning.
  • Clarification loop: planning reveals requirement gaps, so work returns to the spec.
  • Drift loop: verification finds behavior outside the approved artifacts, so code or spec must be corrected.

Self-Improving Knowledge

The knowledge base compounds as features ship. A passing /harness-verify verdict triggers a mandatory post-ship sync: every file in MASTER_INDEX.md is either updated or named with a one-line reason it was untouched. The sweep distills lessons into session-extracts.md candidates, which /context-memory then promotes, defers, or discards. Generic "no updates needed" fails the gate.

1. Ship /harness-verify passes — mechanical gate, alignment, security all green.
2. Sweep Post-ship sync walks every file in MASTER_INDEX.md; updates land where evidence supports them.
3. Triage /context-memory processes candidates: promote (heuristics, PKB, constitution), defer, or discard.
4. Compound The next session's smart routing loads richer memory automatically. KB grows feature by feature.

The Quality Bar

  • Downstream stages do not invent requirements.
  • Implementation is backed by fresh evidence.
  • Traceability from request to proof remains intact.
  • Spec and code stay aligned at the time of merge.

Locked Decisions: Traceability Across Phases

Key Concept: Product and UX decisions recorded in spec.md (including Gray-Area Decisions) flow downstream into planning and implementation. Plan-phase must respect spec decisions; implement-phase must honor both spec and plan decisions without silent reinterpretation.

  • spec-requirements: Records product decisions, user-facing choices, and brownfield preservation rules in spec.md.
  • spec-plan: Treats locked spec decisions as approved inputs. Does not re-open them silently.
  • spec-implement: Enforces Decision Fidelity: follow locked decisions from spec.md without silent reinterpretation.
  • harness-verify: Checks for decision drift: when implementation changes a behavior spec had locked, it's a drift violation.

9. The Verify Gate — Eight-Step Closeout

Verify is the only authority that can move a feature to Done. It runs as an ordered sequence: any failed step reopens tasks and returns the slug to Implementing. The gate is mechanical first, judgmental second, and self-improving last.

Mechanical Gate Run the exact bash commands from plan.md. Exit codes must be 0; evidence must be from this session. No subjective judgment at this step.
Clean-State Check No uncommitted changes outside the feature scope. Working tree matches what the plan authorized.
Alignment Audit Compare every REQ-* in spec.md against delivered code. Detect Missing, Ghost, and Misaligned behavior.
Traceability Audit Walk the full chain: SC → REQ → AC → TASK → CODE → PROOF → REVIEW. Any gap fails the gate.
Security Lens Auth, input validation, secret handling, trust boundaries, prompt-injection defense for AI-facing code.
Optional Fallow Pass Behavior-neutral cleanup if debt was introduced. Only runs when the previous steps flagged it; re-run mechanical proof after cleanup.
Testing Scenarios Walk through testing-scenarios.md — golden path, edge cases, regression risks. Record verdicts in review.md.
Post-Ship Knowledge Sync Run /context-memory sweep. Promote learned heuristics, update the project knowledge base, close the loop. Sync record must be complete before Done.
Verdict Meaning Next Action
Pass All verification steps green. Evidence captured in review.md. Status → Done. Run session handoff.
Fail Failing mechanical gate, spec drift, security gaps, or missing proof. Reopen affected tasks; status → Implementing (or return to spec/planning).

Universal Quality Gates: Anti-Rationalization

Rationalization Reality
"I'll add the tests later." Beyonce Rule: If it's not tested, it doesn't exist.
"The prompt is clear enough." Grilling Phase: Vibe alignment is a trap. Ask 3-5 questions.
"This task is too small to list." Micro-Task Rule: 2-5 min tasks ensure high-fidelity review.
"Naming doesn't matter here." Dictionary Guardian: Shared language is maintainability.

When Blocked, Move Upstream

  • Unclear behavior: return to /spec-research.
  • Weak requirements: refine with /spec-requirements.
  • Design ambiguity: extend /spec-plan with design.md.
  • Weak evidence: rerun implementation validation or use /harness-verify.
  • Context overflow: use /context-session to compact state and evict stale logs before resuming.

Ready for Implement

  • Spec readiness: the user scenarios and acceptance criteria are specific enough that implementation does not need to invent behavior.
  • Task readiness: tasks.md breaks the work into reviewable slices with clear ownership.
  • Design readiness: ambiguous architectural choices are resolved in design.md or ADRs.
  • Validation readiness: each task has an intended proof path before code starts.

Definition of Done

  • Task state, code changes, and fresh validation evidence all agree.
  • The implemented behavior maps back to the approved spec and task IDs.
  • Drift discovered during verification is either removed or promoted upstream into artifacts.
  • A handoff is written if the work stops before the next bounded step is obvious.

Onboarding Entrypoint

When starting a new project, /starter-init detects the repository state and routes the onboarding process accordingly (Greenfield vs Brownfield Mode):

Greenfield Example (new platform module)

1. Capture principles, constraints, and the coding contract with /context-memory.

2. Run /spec-research to explore domain unknowns and constraints.

3. Produce acceptance criteria using /spec-requirements.

4. Draft architecture and boundaries with /visualize and /spec-adr.

5. Convert scope into atomic tasks with /spec-plan.

6. Implement incrementally via /spec-implement and confirm with /harness-verify.

7. End the work window with /context-session and publish continuity in handoff.md.

Brownfield Example (feature change in production system)

1. Start from existing repo context in /context-memory to avoid relearning architecture and coding conventions.

2. Use /spec-research to locate integration points, constraints, and regressions.

3. Define compatibility and rollback criteria in /spec-requirements.

4. Break work into low-risk slices using /spec-plan.

5. Execute one slice at a time with /spec-implement and validate each slice with /harness-verify.

6. Run /harness-verify, including the optional Fallow Pass when cleanup is behavior-neutral.

7. End with /context-session so the next session can resume safely.

  1. Run /context-session before ending a long session.
  2. Write blockers, progress, and the next immediate task into handoff.md.

Multi-Agent Coordination (No-Race Rules)

  • Phase Locking: only one agent edits a locked planning artifact such as spec.md or plan.md at a time.
  • Task Ownership: assign explicit Task IDs from tasks.md when multiple agents implement in parallel.
  • Branch Isolation: each agent works in its own branch; reintegrate only after verification passes.
  • Shared Memory: update the knowledge base after new discoveries so other agents learn instantly.

Brownfield Adoption Rules

  • Do not backfill: write artifacts for the new change, not the entire legacy codebase.
  • Analysis first: in brownfield repositories, always start with /spec-research to find side effects.
  • Gradual memory: promote one durable pattern at a time instead of writing a giant knowledge base up front.

What Not To Do

  • Do not treat chat history as more authoritative than artifacts.
  • Do not skip quality gates when the upstream artifact is weak.
  • Do not claim implementation is complete without fresh evidence.
  • Do not broaden scope silently during implementation.
Proof · Traceability

Proof of Workflow

What Good Looks Like

A useful proposal document needs one concrete walkthrough. This example shows how a request becomes a verified change without relying on chat memory.

Request → Init → Session → Research → Spec → Plan → Implement → Verify → Session End → Harness Lesson?

This proof path intentionally exposes the full harness lifecycle: bootstrap, continuity, delivery, verification, and feedback into harness improvement.

1. Request

Example: "Add social login without breaking existing email login."

The request is still intent, not implementation. The kit does not let coding start here.

2. Spec

Output: spec.md records scenarios, acceptance criteria, and compatibility constraints.

This is where "must not break existing email login" becomes a testable rule instead of a vague preference.

3. Plan

Output: plan.md and tasks.md split the work into schema updates, provider integration, UI changes, and verification tasks.

The implementation no longer decides scope on the fly.

4. Implement

Output: code plus bounded tests mapped back to task IDs.

Each slice is small enough to review without reconstructing the whole feature from chat history.

5. Verify

Output: review.md records whether the feature satisfies the spec and whether the evidence is fresh.

This is where drift is caught before merge instead of after release.

6. Continuity

Output: handoff.md if the work pauses, or clean merge evidence if it finishes.

The next person or next agent can resume without re-discovering intent.

The Traceability Chain

Every shipped change can be walked end-to-end: SC (Success Criteria) → REQ (Requirement) → AC (Acceptance Criterion) → TASK (Task) → CODE (Code Changes) → PROOF (Validation Evidence) → REVIEW (review.md Verdict). The verify gate refuses to close a feature until each link resolves to a concrete artifact in the feature slug folder.

Drift Findings — What Verify Catches

When the alignment audit walks REQ vs. code, it labels every mismatch in one of three categories. Each routes to a different fix.

Missing

A REQ-* exists in spec.md but no code, test, or proof references it. Reopen the matching task or add one.

Ghost

Code or tests exist that no REQ-* authorizes. Either backfill the spec or remove the unauthorized scope.

Misaligned

Both spec and code exist but their behavior diverges — off-by-one, wrong default, missing edge case. Fix code or update spec, then re-prove.

Limitations & What's Next

An honest read of where the kit currently falls short and what's queued for the next cycle. Portfolio readers and adopters deserve the same view I'd give myself in a retro.

Limitation

No formal benchmark suite yet

The kit's verification gates protect each individual feature, but I don't yet measure the kit itself across a controlled set of agent runs. Quality claims are based on personal usage, not on a reproducible eval harness.

Next: seed an eval set of 10–15 representative tasks and score spec quality, drift incidents, and verify-gate accuracy across them.

Limitation

Brownfield discovery is still manual

/spec-research guides the diagnose loop, but mapping integration points and hidden coupling in a large legacy codebase still falls heavily on the human operator. The kit doesn't yet automate test-impact analysis or call-graph extraction.

Next: wire optional codemap and call-graph generation into /spec-research, so brownfield maps include machine-generated boundaries instead of relying on hand-curated lists.

Limitation

Memory promotion is rule-based, not learned

Promotion thresholds (≥800/1200 lines, 3+ subtopics, 5+ refs) are heuristics. They prevent runaway bloat but don't distinguish a high-signal lesson from a verbose one. Some genuinely valuable candidates expire because they didn't accumulate enough cross-references in time.

Next: add an LLM-judged promotion pass during the post-ship sweep, scoring candidates on signal density and uniqueness rather than file-size proxies.

What I'd Build Differently If I Started Over

  • Skill registry as code, not as filenames. The current kit discovers skills via directory layout. A typed registry would let one skill compose another at runtime instead of sequencing them by convention.
  • Spec quality scoring earlier in the loop. The 30-point rubric runs in the evaluator pass; pulling part of it into /spec-requirements itself would catch weak specs before they enter planning.
  • First-class subagent orchestration. Subagent fan-out works, but the orchestration is implicit. A dedicated coordinator would make parallel discovery and adversarial verification more controllable and easier to debug.

12. Maintenance and Governance

Harness Maintenance Workflows

The /harness-maintain command operates in five distinct modes to bootstrap, score, evaluate, diagnose, and continuously improve the harness environment:

Rendering corezero-harness-maintenance…

The Five Execution Modes:

  • Assess Mode (Audit & Score): Scores the repository against the 7-pillar ETCLOVG Taxonomy (Execution, Tools, Context, Logic, Observability, Verification, Governance) using a 1-5 rubric. It checks pending feature candidates (flagging when they exceed 5) and generates the harness-assessment.md report.
  • Create Mode (Bootstrap & Setup): Bootstraps a brand new harness environment for a repository. It runs starter-init, generates the core-policies.md, project-knowledge-base.md, and learned-heuristics.md files, and indexes the project structure.
  • Improve Mode (Observability Triage): Analyzes failures from the auto-tier harness-telemetry.md. It categorizes issues (Mechanical, Alignment, or LLM limits), generates prioritized remediation tasks, applies durable fixes to the owning ETCLOVG pillar, and queues instruction-tier promotions.
  • Eval Mode (Split Evaluation): Runs split evaluations (Mechanical, Alignment, Adversarial/Security, Continuity) over a feature execution flow. It produces eval-report.md and routes gaps to their respective skills (e.g., spec drift to /spec-requirements).

13. FAQ and Common Mistakes

FAQ

Do I need to run all skills for every change?
Not always. Use a ceremony profile. Small changes can run the Tiny profile; risky changes should use the Complex profile.

Can I skip spec and go straight to code?
Only for truly trivial changes. For behavioral changes, skip-spec creates drift risk and weak verification.

What does "done" mean?
Done means task state, code change, and fresh validation evidence all agree.

When should I use handoff?
Before ending any long session, model switch, or context-heavy branch of work.

Common Mistakes

  • Treating chat as source-of-truth instead of artifacts.
  • Marking tasks done before running validation.
  • Mixing unrelated refactors into implementation tasks.
  • Skipping memory promotion for durable findings.
  • Deleting markdown docs before HTML parity is truly stable.

Adoption Pitfalls

  • "The overhead is too high for small fixes." Use the Tiny profile only for truly low-risk work.
  • "Agents are ignoring the skills." Tighten the entrypoint and make the Skill-First rule explicit.
  • "Artifacts are drifting from the code." Make verification a mandatory review gate.
