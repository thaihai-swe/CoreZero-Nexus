About this project Actively maintained

A personal R&D project on agent-first software delivery

I built this kit to answer a question I kept hitting in real work: how do you make AI coding agents behave like disciplined engineers instead of confident chat partners? The result is a 18-skill harness that anchors every change to a locked spec, plans tasks in 2–5 minute units, gates completion behind mechanical verification, and grows a durable memory tier over time so the next session doesn't start from zero.

Everything on this page — the lifecycle, the verify gate, the memory tiers, the failure taxonomy — is implemented in the repo and shipped as installable skills you can drop into any codebase. The diagrams render live from documents/diagrams/*.mmd — the same source of truth used by the kit itself.

Start in 10 Minutes

This is the minimum safe path for a team that wants to trial the kit without reading the entire document first.

1. Pick the target repo From the kit repository, decide which project path will receive the harness files.
2. Install the kit Run curl -fsSL https://raw.githubusercontent.com/thaihai-swe/CoreZero/main/kit/scripts/install.sh | bash -s -- /path/to/project. The installer copies skills/, rules/, scripts/, and seeds the memory router and starter memory.
3. Run /starter-init Bootstrap the harness: capture stack and operating defaults in core-policies.md, seed memory tiers, and run a clean-state check.
4. Create durable context Run /context-memory and refine starter memory files, including the AI coding contract.
5. Start the first feature Run /spec-requirements with a real feature slug and lock acceptance criteria.
Success looks like: a real feature has spec.md, a traceable task breakdown, bounded implementation, and fresh verification evidence before merge.

Priority Rules

These rules override all other guidance when they conflict. They ship in the kit's AGENTS.md.

No flattery, no filler. Start with the answer, action, blocker, or decision. You MUST NOT add ceremonial openers.
Correct false premises. If the user's premise is wrong, you MUST say so before continuing.
Never fabricate. You MUST NOT invent file paths, commit hashes, test results, API names, or repository behavior.
Unknown stays unknown. When information is unavailable, you MUST mark it as [UNKNOWN]. Do not fill gaps with guesses.
Ask only when needed. SHOULD ask before proceeding when ambiguity materially changes the result; otherwise inspect the repo.
Touch only the request. Every changed line MUST directly support the user's request. No drive-by refactors.
Fail loud. You MUST NOT mark work complete if verification was skipped, failed, or only partially run.
Preserve behavior. Existing observable behavior is a contract. You MUST NOT change it unless the user explicitly asks.
Keep design SOLID. SHOULD follow SOLID principles when changing or adding software design.
Apply code-design rules. Check core-zero/policies/code-design.md for cross-cutting pitfalls.
Read the Master Index. SHOULD consult MASTER_INDEX.md before non-trivial work.

3. How It Works (One-Minute Overview)

CoreZero is an engineering harness for AI coding agents. It sits between your request and the code change, routing work through structured skills, persistent artifacts, and mechanical verification gates. Here is the system context in one diagram:

The Core Loop

The harness is organized into five cooperating layers — see Architecture & Repository Shape for the full deep dive. The core delivery loop:

Request → Entrypoint → Skill Contract → Artifacts → Implementation → Verification → Closeout

The entrypoint routes work into a skill. The skill reads and writes artifacts in the feature directory. Implementation is judged against those artifacts instead of chat history. On passing verification, a memory sync promotes durable lessons so the next session starts smarter.

Key Capabilities

Beyond the core loop, four mechanisms prevent the kit from degrading into conversational drift:

Circuit Breaker

Every test/command failure is logged to harness-telemetry.md as an OBS-NNN observation. After 2 consecutive failures on the same task, the agent is locked out of Implementation and forced back to /spec-plan to rethink the approach — no blind guessing toward a passing test.

100% Traceability

Every acceptance criterion in spec.md must map to a task in tasks.md, and every task must have a recorded passing terminal command. One requirement without proof → the build fails.

Three-Track Memory

Instead of dumping the entire codebase into context, the kit loads only what's needed: Instruction tier (absolute constitutional rules), Auto tier (what approaches failed and why), Extracted tier (durable lessons promoted at session end).

Subagent Parallelization

During planning, independent non-blocking components are flagged with [P]. The master agent spawns isolated subagents to build these in parallel, merging results through mechanical gates.

5. Memory & Context Engineering

Three Memory Tiers

Memory is organized into three tiers behind a routing index. The MASTER_INDEX.md router decides which files a session loads by matching task intent against trigger keywords.

Tier Files Owner Lifecycle
Instruction core-policies.md, learned-heuristics.md, project-knowledge-base.md, adr-log.md, core-zero/memories/domain/*, core-zero/project/architecture.md /context-memory + user Promoted from candidates; demoted via CC-* amendment, never silently deleted
Auto harness-telemetry.md telemetry-collector.sh Append-only OBS-* entries; at recurrence-count ≥ 3 proposes promotion to LH-* or CC-*
Extracted artifacts/features/<slug>/session-extracts.md /context-session writes; /context-memory triages EXT-* candidates → promote/defer/discard per triage rules

Flow: session extracts → triage → instruction tier (or discard), with the auto tier capturing failures along the way. Identifier namespaces: CC-* (constitution), LH-* (heuristics), OBS-* (observations), EXT-* (extracts), ADR-* (decisions), INV-* (domain invariants).

Smart Context Routing

Active sessions load only matching memories based on trigger keywords declared in MASTER_INDEX.md. This keeps context windows lean and prevents LLM context fatigue.

Always

core-policies.md — every session.

By Intent: Knowledge

PKB, ADR log, architecture, templates — on architecture, pattern, stack, domain, api surface, constraint.

By Intent: Rules

core-zero/rules/*.md, core-zero/policies/code-design.md — on security, auth, simplicity, ponytail, refactor.

By Intent: Learned

learned-heuristics.md — on recurring, "we always/never", "last time".

By Debug

harness-telemetry.md, prior session-extracts.md — on debug, regression, retro, root cause.

By Domain Packs

core-zero/memories/domain/<name>/* — single keyword match activates the pack.

Memory Promotion Rules & Thresholds

Memory grows under quotas, not free expansion. Two paths feed durable memory:

  • Manual via /context-memory extraction triage: promote (move to instruction tier), defer (needs more evidence), or discard (feature-local or contradicted).
  • Automatic via post-ship sync: passing /harness-verify triggers a sweep of every MASTER_INDEX.md file — each gets updated or named with a one-line reason it was untouched.
Threshold Lines Action
Healthy < 600 No action
Early warning 600–799 Open promotion proposal at promotions.md
Threshold breach 800–1199 Compaction required before new appends; proposal must be reviewed
Hard cap ≥ 1200 Block all appends; split or compact mandatory

Additional triggers: 3+ distinct H2 subtopics → open proposal; 5+ references to the same slice → open proposal. Files on the promotion watchlist are never auto-split — user approval required.

Self-Improving Knowledge Loop

Subagent Fan-Out

Failure Classification

When something goes wrong, classify the cause before patching. Each class routes to a different fix:

Harness Problem

The kit's rails let the agent off — missing gate, stale memory, wrong routing.

Fix: update skills, memory, or verification gates.

Model Problem

The agent had everything it needed but still produced flawed output.

Fix: tighten prompts, add adversarial pass, re-route to capable model.

Spec Problem

Requirements ambiguous, contradictory, or missing.

Fix: return to /spec-requirements, run another grilling wave.

8. The Verify Gate

/harness-verify is the sole authority for marking a feature Done. It closes the loop through 11 steps — mechanical gates, alignment audit, security lens, optional code review, manual testing guide, and post-ship memory sync.

Interactive Verify Gate Flow

A standalone interactive companion visual — click through the gate sequence:

Core Rules

Beyonce Rule

If you liked it, you should have put a test on it. Code not verified unless a mechanical check proves it.

Zero Tolerance

Every AC maps to a task with recorded validation evidence. A single AC with no evidence = Fail. No partial pass.

Anti-Rationalization

Agents cannot skip verification by explaining why it is safe to bypass. Gates are binary and absolute.

Done Authority

This skill is the sole authority for marking a feature Done. No other skill can override the verdict.

Drift Findings — What Verify Catches

The gate classifies each drift it detects into one of three categories:

Missing

A requirement or AC that has no corresponding implementation or proof. Coverage gap.

Ghost

Code, tasks, or complexity not traceable to any spec requirement. Unplanned scope.

Misaligned

Implementation diverges from spec intent. Requires spec amendment or code correction.

The kit detects your repo state and branches accordingly:

Greenfield

New or near-empty repo. Phase B bootstrap: configure policies, seed memories, pick first feature → /spec-requirements. Full greenfield flow:

Brownfield

Existing codebase with history. Phase A archaeology sweep maps boundaries and behaviors. Phase B bootstrap then /spec-research for the first safe feature.

Playbook by Goal

Start a feature

/spec-requirements <slug> — Socratic grilling locks acceptance criteria before any code.

Fix a bug

/spec-research first to find root cause, then /spec-requirements to lock the fix scope.

Document a decision

/spec-adr — capture architectural trade-offs before implementing.

Improve the harness

/harness-maintain improve — reads telemetry, applies durable fixes to the constitution.

When Blocked, Move Upstream

Blocked during implementation? Stop. Do not work around the gap. Move upstream to the skill that owns the uncertainty: /spec-requirements for ambiguous requirements, /spec-plan for design gaps, /spec-research for unknown behavior. A circuit breaker blocks re-entry after 2 consecutive gate failures until /spec-plan re-approves the approach.

  • Context Indexes: read MASTER_INDEX.md first; don't guess file locations.

Golden Configurations

Three principle-driven adapter profiles that match real team setups:

Lightweight Solo (Tiny)

1-2 grilling questions, quick spec, 2-5 task units, minimal session extracts. Ideal for contained features with clear scope.

Standard Team (Standard)

3-5 grilling waves, full spec with NFR-AC binding, 5-15 task units, design document, memory sweep. Default profile.

Complex/Ultra (Complex)

5+ grilling waves, requirements review mandatory, multi-phase plan, circuit breakers, full evaluation report. For high-risk changes.

Discovery Order

When entering a new repository, the agent follows this discovery sequence:

  1. Read AGENTS.md — priority rules and entrypoint
  2. Read MASTER_INDEX.md — context routing and memory locations
  3. Read core-zero/memories/repo/core-policies.md — constitution and guards
  4. Load by-intent memory based on task keywords
  5. Read feature artifacts for the active slug
  6. Read source code JIT as the task requires

Harness Maintenance Workflows

/harness-maintain provides five modes for keeping the kit aligned:

Assess

Score the harness 1-5 across all 7 ETCLOVG subsystems. Produces harness-assessment.md with findings and recommendations.

Create

Bootstrap or rebuild a harness baseline from scratch when none exists.

Improve

Read harness-telemetry.md failure data, classify causes, draft durable fixes. User must approve before promotion.

Eval

Split evaluator architecture. Run feature-level evals and produce eval-report.md. Mandatory before every kit release.

Doctor

10-check self-diagnosis: manifest, sections, paths, thresholds, IDs, ADR status, telemetry. Exit 0 required for release.

Consistency Loop

Every change to a skill, memory, or script must be verified against the manifest and cross-referenced against all other skills that depend on it. Run doctor.sh and install.sh --dry-run after any structural change.

Key checks:

  • Manifest files exist and install.sh resolves all paths
  • No kit/-prefixed paths in skills (shipped paths must be flat)
  • MASTER_INDEX.md routes point to valid file locations
  • Threshold tables are consistent across memory files
  • ADR status vocabulary is clean
  • Telemetry pipeline (collector, count, update, render) is complete
  • Version in manifest.json is the single source of truth

Release Checklist

Before every release:
  1. Run bash kit/scripts/harness/doctor.sh — all 10 checks must pass
  2. Run bash kit/scripts/install.sh /tmp/test --dry-run — verify manifest resolves
  3. Run bash kit/scripts/harness/gate-runner.sh — mechanical gate integrity
  4. Run /harness-maintain eval — full evaluation report
  5. Verify version in kit/manifest.json is bumped (CI auto-bumps on merge)

12. FAQ & Common Mistakes

FAQ

Do I need to install anything besides the kit?

No. The kit is self-contained — skills, scripts, and gates all ship with the installer. You need only the target repository and an AI agent runtime.

Does the kit overwrite my existing files?

No. The installer has a preserve list: core-zero/memories/repo/ and artifacts/ are never overwritten. All other files use overwrite or copyIfMissing strategies explicitly declared in manifest.json.

What happens if I skip a session handoff?

The next session starts without context — no progress, no blockers, no next steps. This is the main source of agent drift. core-policies.md CC-008 mandates handoffs before closing long sessions.

Can I use the kit with any AI model?

Yes, but results vary. The kit provides the structural contract; the model's reasoning capability determines execution quality. Model problems are a separate failure class with dedicated mitigations.

How do I update the kit after installation?

Re-run install.sh on the target repo. The installer re-copies overwrite files and seeds any missing copyIfMissing files, while preserving existing memories and artifacts.

Common Mistakes

  • Skipping intake classification: Every /spec-requirements session must classify the feature (Simple/Moderate/Complex) before grilling. Without it, the grilling depth has no anchor.
  • Design during implementation: /spec-plan requires Part 1 (Technical Design) to complete before any task breakdown. Designing during implementation is the top cause of scope creep.
  • Vibe coding: Implementing without declaring the target task and running a baseline proof command first. This is the #1 violation /spec-implement guards against.
  • Stale evidence: Verifying against old logs instead of running fresh gates. Evidence older than the last code change is not evidence.
  • Generic post-ship sync: A sweep record reading "No updates needed" without per-file reasons is rejected by the gate. Every file gets an explicit verdict.

Honest Limitations

The kit is a personal R&D project. These are the current boundaries:

No formal benchmark suite

Evaluation is qualitative and mode-based (/harness-maintain eval). A systematic benchmark across models and task categories is future work.

Brownfield discovery is still manual

starter-init Phase A archaeology is read-only but not automated. Mapping an existing codebase's boundaries and coupling requires subagent setup and manual review.

Memory promotion is rule-based

Thresholds (600/800/1200 lines, 3+ subtopics, 5+ references) are heuristic. A learned or ML-assisted triage could improve precision for large codebases.

Single-operator workflow

The kit models one agent working one feature at a time. Parallel multi-agent coordination is not yet in scope.

Adoption Pitfalls

  1. Going too fast: Skipping /starter-init Phase B leads to shallow memories that don't ground later sessions.
  2. Over-scoping the first feature: Start with a Simple/Moderate feature. Complex features expose too many unknowns for a team new to the harness.
  3. Ignoring the circuit breaker: Two consecutive gate failures signal a structural issue. Pushing through instead of re-planning compounds the problem.
  4. Treating handoffs as optional: Skipping /context-session END erases context. The next session starts from zero, defeating the continuity model.
